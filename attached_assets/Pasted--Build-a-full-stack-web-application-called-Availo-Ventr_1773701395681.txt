Build a full-stack web application called "Availo Ventry" — a smart visitor management platform for government entities, enterprises, and SMBs. It is an extension product of "Availo" (an employee attendance system by T2 — t2.sa). Ventry = Visit + Entry.

## TECH STACK
- Frontend: Next.js 14 (App Router) with TypeScript, Tailwind CSS, shadcn/ui
- Backend: Next.js API routes + Prisma ORM
- Database: PostgreSQL
- Auth: Role-based with hierarchy (Super Admin → Org Admin → Visitor Manager → Receptionist → Host Employee → Visitor)
- User onboarding: Super Admin creates org + first Org Admin → Org Admin invites internal users → role-specific access
- Language: Bilingual — Arabic (RTL) and English (LTR) with next-intl
- Real-time: Server-Sent Events or WebSockets for live notifications

## CORE CONCEPT
Two visitor registration flows:

### Flow 1: Pre-Registered Visitors (Host-Initiated)
1. An employee (host) logs into the portal and creates a "Visit Request" with a visitor list (name, ID number, phone, purpose, expected date/time).
2. System sends the visitor an SMS/email with a unique QR code or one-time entry pass.
3. When the visitor arrives at reception, the receptionist scans the QR code.
4. System verifies: (a) visitor identity matches, (b) visit is within the valid time window, (c) QR has not been used before (one-time use).
5. Visitor is checked in. Host receives a notification that their guest has arrived.
6. On departure, receptionist checks the visitor out. Pass is invalidated.

### Flow 2: Walk-In / Visitor-Initiated Request
1. Visitor approaches a self-service kiosk or visits a public booking page (unique URL per organization).
2. Visitor fills in: name, ID number, phone, purpose, department/person to visit, preferred date/time.
3. Request enters a queue visible to the Visitor Management team.
4. The request can be approved/rejected via:
   - The web portal (dashboard queue with approve/reject buttons)
   - A Telegram bot (sends request summary → inline buttons: ✅ Approve / ❌ Reject)
5. On approval, visitor receives a QR-based entry pass (SMS/email).
6. Same check-in/check-out flow as Flow 1.

## ROLE HIERARCHY & ADMINISTRATION FLOW

### Role Definitions & Permissions

**1. Super Admin (T2 Platform Level)**
- WHO: T2 staff only. This is NOT per-organization — it's a platform-wide role.
- SCOPE: Access to ALL organizations across the entire Ventry platform.
- CAN DO:
  - Create new organizations (onboard a new client)
  - Set organization subscription tier, license limits (max users, max branches), contract dates
  - Create the FIRST Org Admin user for each organization (initial setup)
  - Activate, suspend, or deactivate organizations
  - View platform-wide analytics (total orgs, total visitors, total requests across all clients)
  - View audit logs across all organizations (for compliance/support)
  - Monitor system health: API response times, Nafath/OTP success rates, error rates
  - Manage platform settings: global rate limits, default verification policies, SMS/OTP provider config
  - Impersonate an Org Admin (support mode — view their dashboard AS them for troubleshooting, with audit trail)
  - Receive platform-wide anomaly alerts (10x traffic spike, new org flooding, etc.)
- CANNOT DO:
  - Approve/reject visit requests (that's org-level)
  - Check in visitors
  - See visitor PII (national IDs) unless explicitly accessing audit logs for compliance
  - Delete organizations permanently (only deactivate — data retention for PDPL)

**2. Org Admin (Organization Level)**
- WHO: The highest authority WITHIN an organization. Created by Super Admin during onboarding.
- SCOPE: Full control of their own organization only. Cannot see other orgs.
- CAN DO:
  - Complete the organization setup: profile, branding, logo, working hours
  - Manage branches (create, edit, deactivate branch locations)
  - Invite and manage ALL user roles within their org (Org Admin, Visitor Manager, Receptionist, Host Employee)
  - Assign users to specific branches or grant access to all branches
  - Set verification policy (Nafath/OTP settings)
  - Set entry mode per branch (staffed/unmanned)
  - Configure all org settings: auto-approve rules, daily caps, flood protection thresholds
  - View and export all reports and analytics for their org
  - Manage blacklist
  - View full audit log for their org
  - Configure Telegram bot integration
  - Set notification preferences per role
  - Activate/deactivate users, reset passwords, force password change
  - Manage visitor pass templates and branding
  - Emergency: trigger evacuation mode, remote gate lockdown
- CANNOT DO:
  - See other organizations' data
  - Change their own subscription tier (that's Super Admin)
  - Access platform-wide settings

**3. Visitor Manager (Organization Level — NEW ROLE)**
- WHO: Staff responsible for managing the visitor approval queue. May also be called "Front Office Manager" or "Security Coordinator."
- SCOPE: Their assigned branch(es) within the org.
- CAN DO:
  - View and manage the visit request queue (approve, reject, reschedule)
  - Approve/reject requests via web portal AND Telegram bot
  - View visitor directory and profiles
  - Manage blacklist (add/remove visitors)
  - View reports for their branch(es)
  - Manage walk-in visitor registration
  - Receive daily digests and pending request notifications
  - View check-in/check-out activity
- CANNOT DO:
  - Invite or manage users (that's Org Admin)
  - Change org settings, verification policy, or entry mode
  - View audit logs or other branches (unless granted multi-branch access)
  - Access billing, subscription, or platform settings

**4. Receptionist (Branch Level)**
- WHO: Front desk staff who physically check visitors in and out.
- SCOPE: Their assigned branch only.
- CAN DO:
  - Scan QR codes and check visitors in/out
  - Search visitors by name, ID, or QR
  - View today's expected visitor list for their branch
  - Verify visitor identity (trigger OTP, visual Nafath check)
  - Print visitor badges
  - Flag overstay visitors
  - Register walk-in visitors (sends to approval queue)
  - View currently checked-in visitors list
- CANNOT DO:
  - Approve or reject visit requests
  - View reports or analytics
  - Manage blacklist
  - Manage users or settings
  - Access other branches

**5. Host Employee (Branch Level)**
- WHO: Any internal employee who invites visitors.
- SCOPE: Their own visitors only. Cannot see other employees' visitors.
- CAN DO:
  - Create visit requests for their own visitors (pre-register a visitor)
  - View their own upcoming and past visitors
  - Receive notifications (visitor arrived, visitor checked out)
  - Cancel their own visit requests
  - Bulk import visitors for their own meetings/events
- CANNOT DO:
  - Approve or reject requests (even their own — approval is by Visitor Manager or Org Admin)
  - Check in visitors
  - View other employees' visitors
  - Access settings, reports, blacklist, or user management

**6. Visitor (External — no portal login)**
- WHO: External person visiting the organization.
- SCOPE: Their own visit only. Interacts via public pages, SMS links, and kiosk.
- CAN DO:
  - Submit a visit request via public booking page
  - View their own entry pass / QR code via the SMS link
  - Self-check-in via phone link (if unmanned entry enabled)
  - Verify identity via OTP or Nafath
  - View status of their own request (pending/approved/rejected)
- CANNOT DO:
  - Log into the portal
  - See any internal data
  - See other visitors

### Super Admin → Organization Onboarding Flow
```
Step 1: Super Admin logs into the T2 Platform Admin Portal
        ↓
Step 2: Super Admin clicks "Create Organization"
        - Fills in: org name (AR + EN), type (govt/enterprise/SMB), address
        - Sets: subscription tier, max users, max branches, contract dates
        - Uploads: org logo (optional, can be done later by Org Admin)
        ↓
Step 3: Super Admin creates the FIRST Org Admin user
        - Fills in: name, email, phone
        - System generates a secure invitation link (valid 72 hours)
        - System sends invitation email + SMS to the new Org Admin:
          "You have been invited as Admin for [Org Name] on Availo Ventry.
           Click here to set up your account: [link]"
        ↓
Step 4: Org Admin clicks the invitation link
        - Sets their password (must meet complexity: 8+ chars, uppercase, number, special)
        - Enables two-factor authentication (mandatory for Org Admin role)
        - Accepts Terms of Service
        ↓
Step 5: Org Admin lands on the "Organization Setup Wizard"
        - Step 5a: Complete org profile (branding, logo, working hours)
        - Step 5b: Set verification policy (Nafath/OTP settings)
        - Step 5c: Create first branch (name, address, entry mode)
        - Step 5d: Invite first users (at least 1 Visitor Manager or Receptionist)
        - Step 5e: Configure Telegram bot (optional, can skip)
        ↓
Step 6: Organization is ACTIVE — ready to receive visitors
```

### Org Admin → User Invitation Flow
```
Step 1: Org Admin goes to Settings → User Management
        ↓
Step 2: Clicks "Invite User"
        - Fills in: name, email, phone
        - Selects role: Org Admin / Visitor Manager / Receptionist / Host Employee
        - Assigns branch: [specific branch] or [All Branches]
        - Optionally sets: department, job title
        ↓
Step 3: System generates invitation (token + link, valid 72 hours)
        - Sends email: "You've been invited to [Org Name] as [Role]. Click to join: [link]"
        - Sends SMS: "You've been invited to Availo Ventry. Check your email."
        ↓
Step 4: Invited user clicks link
        - Sets password
        - Enables 2FA (mandatory for Org Admin & Visitor Manager, optional for others)
        - Accepts Terms of Service
        ↓
Step 5: User account is ACTIVE
        - Redirected to their role-specific dashboard
        - Org Admin sees them in the User Management list as "Active"
        
If invitation expires (72 hours):
        - Org Admin sees the invitation as "Expired" in User Management
        - Can click "Resend" to generate a new token + link (max 3 resends)
        
If Org Admin needs to revoke access:
        - Deactivate user (soft delete — user can't log in, data preserved)
        - Or revoke pending invitation before it's accepted
```

### 13. No-Reception / Unmanned Entry Mode (CRITICAL FEATURE)
Many firms, small businesses, co-working spaces, and branch offices do NOT have a receptionist or front desk. The system must handle visitor verification and entry without any human at the door. The org admin can enable "Unmanned Entry Mode" in settings, which activates the following flows:

**Entry Method A: Smart QR Gate (Primary — for buildings with electronic gate/turnstile/smart lock)**
1. Visitor arrives at the entrance where a mounted tablet or QR scanner is installed next to the gate.
2. Visitor scans their pre-issued QR code on the scanner.
3. System validates: (a) QR is valid and not expired, (b) visit is within scheduled window, (c) QR has not been used before, (d) visitor is not blacklisted.
4. If valid → system triggers a REAL-TIME OTP to the visitor's registered phone number as a second verification factor (proves the person scanning IS the person invited, not someone who screenshot the QR).
5. Visitor enters the OTP on the tablet screen.
6. If OTP matches → gate/lock unlocks automatically (via API integration with smart lock or access control system), visitor is marked as "Checked In", host receives instant notification.
7. If OTP fails 3 times → entry denied, security alert sent to host + org admin, visitor sees "Please contact your host" message with host's phone number.
8. On exit: visitor scans QR again at exit scanner OR taps "Check Out" button on a wall-mounted tablet. If no checkout after threshold hours → auto-alert to host.

**Entry Method B: Phone-Based Self-Check-In (For buildings with NO hardware at door)**
1. When visitor receives their QR entry pass via SMS, the SMS also includes a unique self-check-in link: `https://ventry.t2.sa/checkin/[encrypted-token]`
2. Visitor arrives at the building and opens the link on their own phone.
3. System uses the phone's GPS/location to verify the visitor is physically near the building (geo-fence radius configurable: default 100 meters).
4. If location confirmed → trigger OTP to the same phone (ensures it's the visitor's device, not a forwarded link).
5. Visitor enters OTP → system marks them as "Checked In", host is notified.
6. If GPS is outside geo-fence → entry denied with message: "You must be at the building location to check in."
7. For checkout: visitor clicks "Check Out" from the same link when leaving, or the system auto-checks out after the scheduled window ends.

**Entry Method C: Virtual Reception (Video Call Gate)**
1. Visitor arrives and scans QR on a wall-mounted tablet with a camera.
2. System validates QR → if valid, initiates a video/audio call to the host employee (via web browser, no app needed) OR to a centralized virtual receptionist (one person managing multiple locations remotely).
3. Host/virtual receptionist sees the visitor on camera, verifies identity visually, and taps "Approve Entry" in the app.
4. Gate unlocks / visitor is checked in.
5. If host doesn't answer within 2 minutes → fallback to OTP verification. If OTP also fails → deny entry with "Contact your host" message.

**Entry Method D: Pre-Approved Auto-Entry (For trusted/recurring visitors)**
1. Org admin can flag certain visitors as "Trusted" (e.g., a weekly contractor, a government inspector with ongoing access).
2. Trusted visitors with a valid pre-registered visit only need to scan their QR code — no OTP, no video call, gate opens immediately.
3. System still logs the entry with timestamp, camera snapshot (if available), and marks the source as "Auto-Approved (Trusted)".
4. Trusted status can have an expiry date and can be revoked instantly by org admin.
5. All auto-entries are flagged in audit logs for periodic security review.

## SECURITY REQUIREMENTS (CRITICAL)

### Identity Verification (Configurable per Organization)
The verification method is NOT hardcoded — each organization chooses their own verification policy based on their security needs. Government entities may require Nafath for every visitor, while a small business may only need OTP. The system must support all combinations.

**Integration with Authentica.sa (https://authentica.sa/):**
- OTP verification via SMS (lightweight — visitor receives a code on their phone)
- Nafath verification (Saudi national digital identity — high assurance, government-grade)
- Build the integration layer with mock/stub endpoints that can be swapped with real Authentica.sa API calls
- Nafath flow: System sends a verification request to Authentica → Authentica pushes a notification to the visitor's Nafath app → visitor opens the Nafath app and approves → Authentica confirms to our system → visitor is verified

**Verification Policy Options (org admin configures in Settings → Verification):**
```
Verification Policy:
  ○ None — No identity verification required (visitor registers and gets pass directly)
  ○ OTP Only — Visitor verifies via SMS OTP (default for SMBs)
  ○ Nafath Only — Visitor must verify via Nafath app (high security)
  ○ OTP or Nafath — Visitor chooses their preferred method
  ○ Nafath Required, OTP Fallback — System attempts Nafath first; if visitor doesn't have Nafath app or it fails, fall back to OTP

Apply Verification To:
  ☑ All visitors (every visit requires verification)
  ☑ Walk-in visitors only (pre-registered visitors skip if host vouches)
  ☑ First-time visitors only (returning verified visitors are remembered)
  ☐ High-security zones only (only visitors accessing restricted areas)
  
Nafath Settings (if Nafath is enabled):
  Authentica API Key: [••••••••]

**Verification Flow Logic (implement as middleware/service):**
```typescript
interface VerificationService {
  // Main entry point — checks org policy and routes to correct method
  verifyVisitor(
    orgId: string,
    visitorId: string,
    context: 'pre_registered' | 'walk_in' | 'kiosk' | 'unmanned_entry'
  ): Promise<VerificationResult>;
}

interface VerificationResult {
  verified: boolean;
  method: 'nafath' | 'otp' | 'nafath_with_otp_fallback' | 'trusted_bypass' | 'none';
  verificationId: string;
  nafathTransactionId?: string;
  confidence: 'high' | 'medium' | 'low' | 'none';
  // high = Nafath verified, medium = OTP verified, low = no verification, none = failed
  timestamp: Date;
  failureReason?: string;
}

// Verification decision tree:
// 1. Check org's verification_policy
// 2. Check if context matches nafath_required_for (walk_in, first_visit, all)
// 3. Check if visitor is Trusted AND org allows bypass → skip
// 4. Check if visitor already verified recently (within re-verification window) → skip
// 5. Route to appropriate method:
//    - 'nafath_only' → initiate Nafath, fail if no Nafath
//    - 'otp_only' → send OTP
//    - 'otp_or_nafath' → show visitor a choice screen
//    - 'nafath_required_otp_fallback' → try Nafath, if timeout/fail → OTP
//    - 'none' → auto-pass with confidence: 'none'
```

### Access Control
- Role-based access control (RBAC) enforced on EVERY API route and page — see Permission Matrix in ROLE HIERARCHY section above
- THREE levels of isolation enforced on every request:
  - Level 1 — Authentication: is the user logged in with a valid session?
  - Level 2 — Role check: does the user's role permit this action? (see Permission Matrix)
  - Level 3 — Scope check: is the user accessing data within their org AND their assigned branch?
- Organization-level data isolation (tenant separation) — users can NEVER see other org's data (except Super Admin)
- Branch-level data scoping — Receptionist and Host Employee can only see their own branch's data. Visitor Manager sees their assigned branch(es). Org Admin sees all branches.
- Session management with JWT + refresh tokens, auto-expire after 30 min inactivity
- Two-factor authentication: MANDATORY for Super Admin and Org Admin roles, RECOMMENDED for Visitor Manager, OPTIONAL for Receptionist and Host Employee
- Sensitive actions require re-authentication: changing verification policy, managing users, viewing audit logs, triggering evacuation, impersonating an org
- User invitation flow: users CANNOT self-register — they must be invited by someone with appropriate permission (Super Admin invites Org Admin, Org Admin invites everyone else)
- Account lockout: 5 failed login attempts → lock for 15 minutes → alert the user who invited them
- Password policy: min 8 characters, must include uppercase + number + special character, expire every 90 days (configurable), no reuse of last 5 passwords
- Super Admin impersonation: logged in audit trail with "impersonated_by" field, visible banner in UI, auto-expires after 30 minutes

### Data Security
- All QR codes are one-time-use and time-bound (expire after scheduled window + 30 min grace)
- National ID numbers stored encrypted at rest (AES-256)
- Audit log for every action (who did what, when, from which IP)
- Rate limiting on all public endpoints (booking page, API)
- CAPTCHA on public-facing forms
- SQL injection & XSS protection (Prisma handles parameterized queries)
- CORS restricted to known domains
- PDPL (Saudi Personal Data Protection Law) compliance: data retention limits, consent tracking, right to deletion

### Anti-Flood & Request Abuse Protection (CRITICAL)
The system MUST be resilient to request flooding from both visitor-side and company-side:

**Visitor-Side (Public Booking Page & Kiosk) Protection:**
- IP-based rate limiting: max 5 visit requests per IP per hour, max 20 per IP per day
- National ID rate limiting: max 3 pending requests per national ID across ALL organizations at any time
- Phone number rate limiting: max 5 OTP requests per phone per hour (prevents SMS bombing / cost attack)
- Progressive CAPTCHA: invisible reCAPTCHA on first request → visible challenge after 2nd request from same session → block after 5 failed CAPTCHAs
- Honeypot fields in the public booking form (hidden fields that bots fill but humans don't — auto-reject if filled)
- Request cooldown: same visitor cannot submit a new request to the same organization within 30 minutes of a previous submission
- Disposable/VoIP phone number detection: flag requests using known disposable number patterns
- Geo-fencing (optional): org admin can restrict visit requests to originate from within Saudi Arabia only
- Abuse scoring: each request gets a risk score (0-100) based on: IP reputation, request velocity, form completion speed (too fast = bot), duplicate patterns. Requests scoring above threshold are auto-held for manual review instead of entering the normal queue.

**Company-Side (Host Employee & Internal) Protection:**
- Per-employee daily request cap: configurable limit (default 20 visit requests per employee per day) — prevents a compromised or rogue account from flooding the system
- Per-organization daily cap: configurable maximum total requests per org per day (e.g., 200 for SMB, 1000 for enterprise, 5000 for government) — exceeding triggers admin alert, does not hard-block
- Bulk import size limit: max 500 visitors per CSV upload, max 3 bulk imports per day per user
- Session anomaly detection: if an employee account suddenly creates 50+ requests in 10 minutes (vs their normal 2-3/day), auto-lock the account and alert the org admin
- API key rate limiting: if the system exposes APIs for integration, enforce per-key request quotas (e.g., 100 requests/minute) with exponential backoff on 429 responses
- Approval queue overflow protection: if pending requests exceed a threshold (e.g., 500 per org), new requests are queued in a "waiting room" and the org admin is alerted — prevents the approval queue from becoming unmanageable

**Global Platform Protection:**
- WAF (Web Application Firewall) rules: block known bot user agents, enforce request size limits
- DDoS protection at infrastructure level (Cloudflare or equivalent)
- Global rate limit: max 1000 requests per minute per organization across all endpoints
- Suspicious pattern alerts to T2 super admin: org with 10x normal traffic, new org creating thousands of requests on day one, etc.
- Automatic temporary IP ban after 100 failed requests in 5 minutes (with CAPTCHA unlock)
- All rate limit hits logged in audit trail with IP, user agent, and trigger reason

**Database Model Addition — Request Throttle Tracking:**

Rate Limit Log:

```

**Admin Controls for Flood Protection:**
- Org admin can configure all thresholds (daily caps, cooldown periods, abuse score threshold)

- One-click IP ban from the flagged requests view
- Whitelist for trusted IPs/partners (e.g., a partner company's office IP skips stricter rate limits)
- Alert notification channel: email + Telegram to security team when flood is detected

#### 8. Accessibility Notes
```
All color pairings MUST meet WCAG 2.1 AA contrast ratios (4.5:1 for text, 3:1 for large text/UI):
  ✅ navy-900 on white = 16.75:1 (AAA)
  ✅ brand-600 on white = 4.56:1 (AA)
  ✅ brand-700 on white = 5.96:1 (AA)
  ✅ success-700 on success-50 = 6.2:1 (AA)
  ✅ danger-700 on danger-50 = 6.8:1 (AA)
  ✅ warning-700 on warning-50 = 5.1:1 (AA)
  ⚠️ brand-600 on brand-50 = 4.3:1 (borderline — use brand-700 for small text on light blue)
  ⚠️ warning-500 on white = 3.2:1 (fails for small text — use warning-600 or 700 for text)

  Never use:
  ❌ Light text on light background
  ❌ Brand-400 or lighter for text on white (fails contrast)
  ❌ Status colors alone to convey meaning — always pair with icon + text label
```

RBAC MIDDLEWARE (CRITICAL — implement before any page):
```typescript
// /lib/rbac/permissions.ts
// Every API route and page MUST check:
// 1. Is user authenticated?
// 2. Does user's role have permission for this action?
// 3. Is user accessing data within their org? (tenant isolation)
// 4. Is user accessing data within their branch? (branch scoping)

// Example middleware:
function requireRole(...roles: Role[]) {
  return (req, res, next) => {
    if (!roles.includes(req.user.role)) return res.status(403);
    if (req.user.role !== 'super_admin' && req.params.orgId !== req.user.orgId)
      return res.status(403); // tenant isolation
    next();
  };
}
```
